## 1. 개요 (Overview)

### 1.1 제품 비전

ContextSync는 소규모 개발팀(2~10명)이 Claude Code 및 Claude AI를 활용한 작업에서 발생하는 **프롬프트, AI 응답, 실행 계획(Plan)** 을 중앙에서 아카이빙하고, 팀원 간 **작업 진행 상황 공유** 및 **컨텍스트 충돌 감지**를 지원하는 SaaS 웹 애플리케이션이다.

현재 Claude Code의 Agent Teams 기능은 단일 세션 내 멀티 에이전트 협업에 초점을 맞추고 있지만, **세션 간·팀원 간 컨텍스트의 영속적 관리와 충돌 탐지**는 지원하지 않는다. ContextSync는 이 간극을 메운다.

### 1.2 문제 정의

| 문제 | 설명 |
|------|------|
| **컨텍스트 유실** | 각 팀원이 Claude와 나눈 대화가 개인 세션에 갇혀, 팀 전체의 의사결정 맥락이 소실된다. |
| **작업 충돌** | 두 명 이상이 동일 모듈에 대해 서로 다른 지시를 내려 상충하는 코드/플랜이 생성된다. |
| **진행 상황 불투명** | "누가 어떤 작업을 AI에게 시켰고, 결과가 어땠는지" 실시간 파악이 불가능하다. |
| **지식 재활용 부재** | 과거 효과적이었던 프롬프트나 AI 플랜을 팀 차원에서 재사용하기 어렵다. |

### 1.3 목표 사용자

- **소규모 개발팀 (2~10명)**: 스타트업, 사이드 프로젝트, 에이전시 등
- Claude Code 또는 Claude AI를 일상 개발 워크플로에 적극 활용하는 팀
- Git 기반 협업에 익숙하며, AI 기반 개발에 적극적인 엔지니어

## 2. 목표 및 성공 지표 (Goals & Success Metrics)

### 2.1 제품 목표

1. 팀원 간 AI 작업 컨텍스트의 **완전한 투명성** 확보
2. 컨텍스트 충돌을 **사전 감지**하여 불필요한 재작업 방지
3. 효과적인 프롬프트와 플랜의 **팀 자산화**

### 2.2 핵심 성과 지표 (KPIs)

| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 컨텍스트 충돌로 인한 재작업 감소율 | 60% 이상 감소 | 도입 전후 충돌 발생 건수 비교 |
| 팀 컨텍스트 파악 소요 시간 | 5분 이내 | 신규 세션 시작 시 팀 현황 파악까지 걸리는 시간 |
| 프롬프트/플랜 재사용률 | 월 30% 이상 | 아카이브에서 재활용된 프롬프트 비율 |
| 주간 활성 사용률 (WAU) | 팀원의 80% 이상 | 주 1회 이상 대시보드 접속 비율 |

## 3. 사용자 페르소나 (User Personas)

### 페르소나 1: 테크리드 민수

- **역할:** 4인 스타트업 CTO
- **상황:** Claude Code Agent Teams로 백엔드/프론트엔드 동시 개발 중
- **Pain Point:** 팀원 각자가 Claude에게 지시한 아키텍처 방향이 충돌하여, 코드 머지 시점에서야 문제를 발견
- **기대:** 실시간으로 팀원들의 AI 작업 현황과 의사결정을 조망하고 싶다

### 페르소나 2: 주니어 개발자 지은

- **역할:** 입사 2개월차 프론트엔드 개발자
- **상황:** 시니어가 Claude에게 내린 설계 지시와 그 결과를 참조하여 구현 작업 수행
- **Pain Point:** 시니어의 컨텍스트를 파악하려면 직접 물어봐야 해서 비효율적
- **기대:** 과거 AI 대화 기록과 플랜을 검색하여 스스로 컨텍스트를 획득하고 싶다

### 페르소나 3: 프리랜서 영호

- **역할:** 3개 클라이언트 프로젝트를 병행하는 풀스택 개발자
- **상황:** 프로젝트별로 Claude와 나눈 대화가 흩어져 있음
- **Pain Point:** 프로젝트 간 컨텍스트 스위칭 시 이전 맥락을 잃어버림
- **기대:** 프로젝트별로 AI 작업 히스토리를 정리하고, 협업자와 공유하고 싶다

## 4. 핵심 기능 (Core Features)

### 4.1 세션 아카이브 & 히스토리

프롬프트, AI 응답, 생성된 플랜을 자동으로 수집·저장하는 핵심 기능.

**기능 상세:**
- Claude Code CLI 세션, Claude.ai 웹 대화, API 호출 기록 수집
- 각 세션에 자동 태깅: 프로젝트, 모듈, 파일 경로, 작성자
- 전문 검색(Full-text Search) 및 필터링 (날짜, 작성자, 프로젝트, 태그)
- 세션 간 연결 관계 시각화 (이 세션이 어떤 이전 세션의 연장인지)
- 북마크 & 핀 기능으로 중요 대화 고정

**데이터 수집 방식:**
- **Claude Code:** CLI Hook(실험적 기능)을 활용한 자동 수집 또는 수동 Export → Import
- **Claude.ai:** 브라우저 확장 프로그램 통한 수집 또는 API 연동
- **API 사용자:** Webhook/이벤트 기반 자동 수집

### 4.2 실시간 팀 대시보드

팀원들의 AI 작업 현황을 한눈에 파악하는 대시보드.

**기능 상세:**
- 팀원별 현재 활성 세션 목록 및 작업 중인 파일/모듈 표시
- 타임라인 뷰: 시간순으로 팀 전체의 AI 작업 흐름 표시
- 칸반 뷰: 프로젝트/모듈별 작업 현황 보드
- 실시간 알림: 특정 파일/모듈에 새로운 AI 작업이 발생했을 때 통지
- 일일/주간 요약 리포트 자동 생성

### 4.3 컨텍스트 충돌 감지 (Context Conflict Detection)

동일 리소스에 대한 상충하는 AI 지시/결과를 사전에 탐지하는 기능. ContextSync의 핵심 차별화 요소.

**충돌 유형:**
1. **파일 충돌:** 두 명 이상이 같은 파일에 대한 수정을 AI에게 지시
2. **설계 충돌:** 동일 모듈에 대해 상반된 아키텍처/패턴이 제안됨
3. **의존성 충돌:** 서로 다른 라이브러리 버전이나 기술 스택이 지정됨
4. **계획 충돌:** 동일 기능에 대한 상반된 구현 계획이 병행 진행 중

**감지 메커니즘:**
- 파일 경로, 모듈명, 함수명 기반 자동 매칭
- AI 기반 의미론적 유사도 분석 (두 플랜의 방향이 상충하는지)
- Git branch 정보와 연동하여 머지 충돌 사전 예측
- 충돌 심각도 레벨링: `Info` → `Warning` → `Critical`

**충돌 해소 워크플로:**
- 충돌 감지 시 관련 팀원에게 실시간 알림
- 충돌 상세 비교 뷰 (Side-by-side Diff)
- 인라인 코멘트 및 토론 스레드
- 해결 방향 합의 후 "Resolved" 마킹

### 4.4 프롬프트 & 플랜 라이브러리

팀에서 검증된 프롬프트와 AI 생성 플랜을 재사용 가능한 자산으로 관리.

**기능 상세:**
- 효과적인 프롬프트를 템플릿으로 저장
- 플랜에 대한 팀원 리뷰 & 평점
- 카테고리별 분류: 코드리뷰, 아키텍처 설계, 디버깅, 테스트 작성 등
- 프롬프트 변수화: `{{module_name}}`, `{{tech_stack}}` 등 동적 삽입 지원
- 버전 관리: 프롬프트 개선 이력 추적

### 4.5 프로젝트 컨텍스트 맵

프로젝트의 AI 활용 현황을 시각적으로 조망하는 맵.

**기능 상세:**
- 코드베이스 구조 위에 AI 활동을 오버레이하여 시각화
- 모듈별 AI 작업 빈도, 최근 변경, 담당자 표시
- 의존 관계 그래프: 모듈 간 연결과 영향 범위 시각화
- "Hot Zone" 표시: AI 작업이 집중되는 영역 하이라이트

## 5. 사용자 시나리오 (User Scenarios)

### 시나리오 1: 아침 스탠드업 대체

> 민수(테크리드)가 출근 후 ContextSync 대시보드를 열면, 어젯밤 팀원들이 Claude에게 지시한 작업과 결과가 타임라인으로 정리되어 있다. 지은이가 로그인 모듈 리팩토링을 Claude에게 지시했고, 영호가 같은 모듈의 API 엔드포인트를 변경하는 플랜을 받은 것이 `Warning` 충돌로 감지되어 있다. 민수는 충돌 비교 뷰를 열어 두 플랜의 차이를 확인하고, 코멘트를 남겨 조율한다.

### 시나리오 2: 신규 팀원 온보딩

> 새로 합류한 개발자가 ContextSync에서 프로젝트 컨텍스트 맵을 열어 각 모듈의 AI 활용 이력을 탐색한다. 결제 모듈에 대한 시니어의 아키텍처 설계 대화를 읽고, 검증된 프롬프트 템플릿을 활용하여 관련 기능 구현 작업을 시작한다.

### 시나리오 3: 스프린트 회고

> 팀이 주간 요약 리포트를 통해 이번 스프린트에서 발생한 컨텍스트 충돌 3건을 확인한다. 충돌 원인을 분석하여, 모듈 담당자를 명확히 하고, 관련 컨벤션을 프롬프트 라이브러리에 추가한다.

## 7. 데이터 수집 전략 (Data Ingestion Strategy)

### 7.1 Claude Code 연동

Claude Code는 현재 Hooks 시스템을 제공하며, 이를 통해 세션 이벤트를 캡처할 수 있다.

**접근 방식:**
1. **Phase 1 (MVP):** Claude Code 세션 종료 시 JSONL 로그를 Export → ContextSync에 수동/반자동 Import
2. **Phase 2:** Claude Code Hook을 통해 세션 시작/종료/메시지 이벤트를 Webhook으로 자동 전송
3. **Phase 3:** 공식 Claude Code Extension/Plugin으로 네이티브 통합

### 7.2 Claude.ai 웹 연동

**접근 방식:**
1. **Phase 1 (MVP):** 브라우저 확장 프로그램으로 대화 내용 캡처 및 전송
2. **Phase 2:** Anthropic API 활용 시 Middleware 레이어로 자동 수집

### 7.3 데이터 프라이버시

- 모든 세션 데이터는 팀 단위로 격리 (Tenant Isolation)
- 개인 세션 중 팀 공유 원하지 않는 대화는 "Private" 설정 가능
- 민감 정보 자동 마스킹 옵션 (API 키, 비밀번호 등)
- 데이터 보존 기간 설정 (30일 / 90일 / 무제한)
- GDPR 및 개인정보보호법 준수

```
         │
         ▼
┌─────────────────┐
│  4. 심각도       │  Info / Warning / Critical 분류
│     판정         │  관련 팀원에게 알림 전송
└─────────────────┘
```

### 8.2 충돌 분석 프롬프트 (예시)

```
다음 두 개의 AI 세션에서 생성된 변경 계획을 분석하여 
충돌 여부를 판단해주세요.

[세션 A - 작성자: 지은]
{session_a_plan}

[세션 B - 작성자: 영호]
{session_b_plan}

다음 기준으로 분석:
1. 동일 파일/함수에 대한 상충하는 수정이 있는가?
2. 아키텍처 또는 설계 방향이 서로 호환되는가?
3. 의존성(라이브러리, API) 선택이 일관되는가?
4. 충돌 심각도는? (info / warning / critical)

JSON 형식으로 응답해주세요.
```

## 9. UI/UX 핵심 화면 (Key Screens)

### 9.1 팀 대시보드 (메인)

- 상단: 프로젝트 선택기 + 팀원 아바타 (온라인/작업중 상태)
- 중앙: 타임라인 뷰 (세로 스크롤, 시간순 AI 활동 피드)
- 우측 사이드바: 활성 충돌 목록 (심각도별 색상 코딩)
- 하단: 오늘의 통계 (세션 수, 충돌 건수, 활성 플랜)

### 9.2 세션 상세 뷰

- 프롬프트 ↔ 응답 대화 스레드 형식
- 각 메시지에 태그, 코멘트, 북마크 추가 가능
- "플랜으로 추출" 버튼: AI 응답에서 실행 계획만 분리 저장
- 관련 세션 링크 (이전/이후 연결된 세션)

### 9.3 충돌 비교 뷰

- Git Diff 스타일의 Side-by-side 비교
- 충돌 지점 하이라이트
- 인라인 토론 스레드
- "Resolve" 버튼으로 해결 상태 변경

### 9.4 프롬프트 라이브러리

- 카드형 그리드 레이아웃
- 카테고리 필터, 인기도/최신순 정렬
- 원클릭 복사 및 변수 커스터마이징

## 10. 개발 로드맵 (Roadmap)

### Phase 1: MVP (8주)

**목표:** 핵심 가치 검증 — "팀의 AI 작업이 보인다"

| 주차 | 마일스톤 |
|------|---------|
| 1-2주 | 프로젝트 셋업, 인증, 데이터 모델 설계 및 구현 |
| 3-4주 | 세션 Import(수동 업로드), 세션 목록/상세 뷰 |
| 5-6주 | 팀 대시보드(타임라인 뷰), 기본 검색 |
| 7-8주 | 규칙 기반 충돌 감지(파일 경로 기반), 알림(이메일/Slack) |

**MVP 포함 기능:**
- GitHub OAuth 로그인
- 프로젝트/팀 생성 및 관리
- 세션 수동 업로드 (JSON/Markdown)
- 세션 목록, 상세, 검색
- 타임라인 대시보드
- 파일 경로 기반 충돌 감지
- Slack/이메일 알림

### Phase 2: Core (6주)

**목표:** 자동화 및 지능형 충돌 감지

- 브라우저 확장 프로그램 (Claude.ai 대화 자동 수집)
- Claude Code Hook 연동 (자동 세션 수집)
- AI 기반 의미론적 충돌 분석
- 프롬프트 라이브러리 v1
- 프로젝트 컨텍스트 맵 (기본)
- 실시간 WebSocket 대시보드

### Phase 3: Growth (6주)

**목표:** 팀 생산성 극대화

- 프롬프트 템플릿 변수화 및 버전 관리
- 주간/스프린트 AI 활용 리포트
- Git 연동 (Branch ↔ 세션 매핑)
- 칸반 뷰
- API 공개 (외부 도구 연동)
- 팀 온보딩 가이드 자동 생성

## 12. 리스크 및 완화 전략 (Risks & Mitigations)

| 리스크 | 영향도 | 확률 | 완화 전략 |
|--------|--------|------|----------|
| Anthropic API/플랫폼 정책 변경 | 높음 | 중간 | 다중 LLM 지원 아키텍처로 설계, API 추상화 레이어 |
| Claude Code Hook API 불안정 | 중간 | 높음 | 수동 Import를 항상 Fallback으로 유지, Export 기반 수집 병행 |
| 데이터 프라이버시 우려 | 높음 | 중간 | 셀프호스팅 옵션 제공, E2E 암호화, 민감정보 자동 마스킹 |
| 낮은 초기 채택률 | 높음 | 중간 | 기존 워크플로 침습 최소화, Slack Bot 연동으로 진입장벽 낮춤 |
| 충돌 감지 오탐 (False Positive) | 중간 | 높음 | 초기엔 보수적 임계값, 사용자 피드백 루프로 지속 개선 |

## 부록 A: 용어 정의

| 용어 | 정의 |
|------|------|
| **세션 (Session)** | 사용자와 Claude 간의 하나의 연속된 대화 단위 |
| **플랜 (Plan)** | AI가 생성한 구현/실행 계획 (코드 변경 계획, 아키텍처 제안 등) |
| **컨텍스트 충돌** | 동일 리소스(파일, 모듈, API 등)에 대해 상충하는 AI 작업이 병행되는 상황 |
| **프롬프트 템플릿** | 변수를 포함하여 재사용 가능하도록 구조화된 프롬프트 |
| **컨텍스트 맵** | 프로젝트 코드베이스 위에 AI 활동을 시각적으로 표현한 오버레이 |
