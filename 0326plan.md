# 팀 구성원 및 데이터 삭제 기능 기획

> **작성일:** 2026-03-26
> **상태:** Reviewed (교차검수 완료)
> **대상 모듈:** `projects`, `sessions`, `prd-analysis`, `ai-evaluation`, `activity`

---

## 1. 배경 및 문제 정의

### 현재 상태

- Owner는 `DELETE /projects/:projectId/collaborators/:userId` 로 구성원을 제거할 수 있음
- 그러나 **collaborator 레코드만 삭제**되고, 해당 구성원이 생성한 데이터는 모두 잔존
- 잔존 데이터: 세션, 메시지, PRD 문서/분석, AI 평가, 활동 로그, 프롬프트 템플릿, 충돌 기록, 동기화 세션

### 현재 구현의 추가 문제점 (교차검수에서 발견)

1. **Owner 자기 삭제 방지가 UI에만 존재** — 프론트엔드에서 버튼을 숨기지만, API 레벨에서는 검증하지 않음. 직접 API 호출 시 Owner 자신을 제거할 수 있는 취약점
2. **확인 절차 없음** — 현재 UI에서 버튼 클릭 즉시 삭제 mutation 실행 (모달/확인 없음)

### 목표

- Owner가 특정 구성원을 **프로젝트에서 제거**하면서 해당 구성원의 **데이터도 함께 삭제**할 수 있는 옵션 제공
- 기존 "구성원만 제거" 동작도 유지 (하위 호환)
- 삭제 전 **확인 단계**를 통해 실수로 인한 데이터 손실 방지
- Owner 자기 삭제 방지를 **서버 단에서도 검증**

## 2. FK CASCADE 분석 (교차검수 기반)

실제 마이그레이션 파일을 확인하여 정리한 ON DELETE 동작:

### CASCADE 관계 (부모 삭제 시 자동 삭제)

```
sessions 삭제 시 자동 CASCADE:
├── messages (session_id → ON DELETE CASCADE)
├── synced_sessions (session_id → ON DELETE CASCADE)
├── conflicts (session_a_id → ON DELETE CASCADE)
└── conflicts (session_b_id → ON DELETE CASCADE)

prd_documents 삭제 시 자동 CASCADE:
├── prd_analyses (prd_document_id → ON DELETE CASCADE)
│   └── prd_requirements (prd_analysis_id → ON DELETE CASCADE)

ai_evaluations 삭제 시 자동 CASCADE:
├── ai_evaluation_dimensions (evaluation_id → ON DELETE CASCADE)
│   └── ai_evaluation_evidence (dimension_id → ON DELETE CASCADE)
```

### SET NULL 관계 (부모 삭제 시 NULL로 변경)

```
prompt_templates.project_id → ON DELETE SET NULL
ai_evaluation_evidence.message_id → ON DELETE SET NULL
ai_evaluation_evidence.session_id → ON DELETE SET NULL
```

### ON DELETE 규칙 없음 (주의 필요)

```
prompt_templates.author_id → NO DELETE RULE (NOT NULL) ⚠️
conflicts.resolved_by → NO DELETE RULE (nullable)
conflicts.reviewer_id → NO DELETE RULE (nullable)
```

> **핵심:** `prompt_templates.author_id`는 NOT NULL이면서 ON DELETE 규칙이 없음. 사용자 삭제 시 FK violation이 발생하나, 이 기능에서는 user 레코드를 삭제하지 않으므로 문제 없음. 다만, prompt_templates 자체를 삭제할 때는 직접 DELETE 쿼리 필요.

## 3. 기능 요구사항

### 3.1 API — 구성원 제거 (확장)

기존 `DELETE /projects/:projectId/collaborators/:userId` 엔드포인트에 쿼리 파라미터 추가:

```
DELETE /projects/:projectId/collaborators/:userId?deleteData=true
```

| 파라미터     | 타입    | 기본값  | 설명                                               |
| ------------ | ------- | ------- | -------------------------------------------------- |
| `deleteData` | boolean | `false` | `true`이면 구성원의 프로젝트 내 데이터도 함께 삭제 |

**권한:** `collaborator:manage` (Owner 전용, 기존과 동일)

**제약 조건:**

- Owner 자신은 삭제 불가 — **서버 단 검증 추가** (400 Bad Request)
- 삭제 대상이 프로젝트 구성원이 아닌 경우 404

### 3.2 삭제 대상 및 실행 순서 (CASCADE 최적화)

`deleteData=true` 시, CASCADE를 활용하여 **명시적으로 삭제해야 하는 테이블만** 직접 삭제:

| 순서 | 명시적 삭제             | 조건                                                 | CASCADE로 자동 삭제되는 항목                        |
| ---- | ----------------------- | ---------------------------------------------------- | --------------------------------------------------- |
| 1    | `sessions`              | `user_id = userId AND project_id = projectId`        | → messages, synced_sessions, conflicts(session_a/b) |
| 2    | `prd_documents`         | `user_id = userId AND project_id = projectId`        | → prd_analyses → prd_requirements                   |
| 3    | `ai_evaluations`        | `target_user_id = userId AND project_id = projectId` | → ai_evaluation_dimensions → ai_evaluation_evidence |
| 4    | `activity_log`          | `user_id = userId AND project_id = projectId`        | —                                                   |
| 5    | `prompt_templates`      | `author_id = userId AND project_id = projectId`      | —                                                   |
| 6    | `project_invitations`   | `inviter_id = userId AND project_id = projectId`     | —                                                   |
| 7    | `project_collaborators` | `user_id = userId AND project_id = projectId`        | —                                                   |

> **참고:** 기존 기획에서 13개 테이블을 명시적 삭제 대상으로 잡았으나, CASCADE 분석 결과 **7개 테이블만 직접 삭제**하면 충분. messages, synced_sessions, conflicts, prd_analyses, prd_requirements, ai_evaluation_dimensions, ai_evaluation_evidence는 CASCADE로 자동 처리.

> **참고:** `users` 테이블의 레코드는 삭제하지 않음. 사용자는 다른 프로젝트에서 활동 중일 수 있으므로 프로젝트 범위 데이터만 삭제.

### 3.3 conflicts 잔존 참조 처리

구성원의 세션 삭제 시 관련 conflicts는 CASCADE로 자동 삭제됨. 단, 다른 구성원의 충돌에서 `resolved_by`/`reviewer_id`로 참조되는 경우:

- `resolved_by = userId` → **NULL로 업데이트** (삭제 전)
- `reviewer_id = userId` → **NULL로 업데이트** (삭제 전)
- 이 두 컬럼은 nullable이므로 안전

#### 핵심 구현: 트랜잭션 기반 삭제

```typescript
// collaborator.repository.ts
export async function deleteCollaboratorData(
  db: Db,
  projectId: string,
  userId: string,
): Promise<DeletedDataSummary> {
  return db.transaction().execute(async (trx) => {
    // 0. 삭제 전 카운트 조회 (활동 로그용)
    const counts = await getCollaboratorDataCounts(trx, projectId, userId);

    // 1. conflicts에서 resolved_by/reviewer_id 참조 NULL 처리
    await trx
      .updateTable('conflicts')
      .set({ resolved_by: null })
      .where('project_id', '=', projectId)
      .where('resolved_by', '=', userId)
      .execute();

    await trx
      .updateTable('conflicts')
      .set({ reviewer_id: null })
      .where('project_id', '=', projectId)
      .where('reviewer_id', '=', userId)
      .execute();

    // 2. sessions 삭제 (CASCADE: messages, synced_sessions, conflicts)
    await trx
      .deleteFrom('sessions')
      .where('project_id', '=', projectId)
      .where('user_id', '=', userId)
      .execute();

    // 3. prd_documents 삭제 (CASCADE: prd_analyses → prd_requirements)
    await trx
      .deleteFrom('prd_documents')
      .where('project_id', '=', projectId)
      .where('user_id', '=', userId)
      .execute();

    // 4. ai_evaluations 삭제 (CASCADE: dimensions → evidence)
    await trx
      .deleteFrom('ai_evaluations')
      .where('project_id', '=', projectId)
      .where('target_user_id', '=', userId)
      .execute();

    // 5. activity_log 삭제
    await trx
      .deleteFrom('activity_log')
      .where('project_id', '=', projectId)
      .where('user_id', '=', userId)
      .execute();

    // 6. prompt_templates 삭제
    await trx
      .deleteFrom('prompt_templates')
      .where('project_id', '=', projectId)
      .where('author_id', '=', userId)
      .execute();

    // 7. project_invitations 삭제
    await trx
      .deleteFrom('project_invitations')
      .where('project_id', '=', projectId)
      .where('inviter_id', '=', userId)
      .execute();

    // 8. collaborator 레코드 삭제
    await trx
      .deleteFrom('project_collaborators')
      .where('project_id', '=', projectId)
      .where('user_id', '=', userId)
      .execute();

    return counts;
  });
}
```

#### Owner 자기 삭제 방지 (서버 단 추가)

```typescript
// project.service.ts — removeCollaborator() 내부
export async function removeCollaborator(
  db: Db,
  projectId: string,
  userId: string,
  targetUserId: string,
  deleteData: boolean = false,
): Promise<void> {
  if (userId === targetUserId) {
    throw new BadRequestError('Cannot remove yourself from the project');
  }

  const project = await projectRepo.findProjectById(db, projectId);
  if (!project) throw new NotFoundError('Project');

  await assertPermission(db, projectId, userId, 'collaborator:manage');

  // 대상이 실제 구성원인지 확인
  const targetRole = await getUserRoleInProject(db, projectId, targetUserId);
  if (!targetRole) throw new NotFoundError('Collaborator');

  if (deleteData) {
    await collabRepo.deleteCollaboratorData(db, projectId, targetUserId);
    logActivity(db, {
      projectId,
      userId,
      action: 'collaborator_removed',
      entityType: 'collaborator',
      entityId: targetUserId,
      metadata: { deleteData: true, ...counts },
    });
  } else {
    await collabRepo.removeCollaborator(db, projectId, targetUserId);
    logActivity(db, {
      projectId,
      userId,
      action: 'collaborator_removed',
      entityType: 'collaborator',
      entityId: targetUserId,
    });
  }
}
```

## 5. 엣지 케이스 및 고려사항

### 5.1 Owner 자기 삭제 방지

- **현재:** UI에서만 버튼 숨김 → API 직접 호출 시 취약
- **수정:** 서버 단에서 `userId === targetUserId` 검증 추가 (400 Bad Request)

### 5.2 대량 데이터 삭제 시 성능

- CASCADE 활용으로 쿼리 수가 13개 → 7개로 감소
- 세션이 수천 건이라도 `DELETE FROM sessions WHERE ...`는 단일 쿼리로 처리 (PostgreSQL이 인덱스 활용)
- `sessions.user_id`와 `sessions.project_id` 모두 인덱싱 되어 있으므로 성능 이슈 낮음
- 극단적 케이스 대비: 트랜잭션 statement_timeout 30초 설정

### 5.3 동시성

- collaborator 삭제를 트랜잭션 **마지막**에 수행 → 삭제 진행 중에도 구성원의 API 접근이 가능하나, 트랜잭션이 커밋되면 일괄 적용
- 트랜잭션 격리 수준(READ COMMITTED) 기본값으로 충분

### 5.4 conflicts 참조 무결성

- **session FK (CASCADE):** 구성원 세션이 참여한 충돌은 세션 삭제 시 자동 삭제
- **resolved_by / reviewer_id (NO DELETE RULE):** 구성원이 다른 충돌을 resolve/review한 경우, user 레코드는 남아있으므로 FK violation 없음. 단, 프로젝트에서 제거된 사용자가 resolver로 표시되는 것이 적절한지는 비즈니스 판단 필요 → v1에서는 NULL로 업데이트

### 5.5 AI 평가 — triggered_by vs target

- `target_user_id = userId`: 해당 구성원에 대한 평가 → **삭제**
- `triggered_by_user_id = userId`: 해당 구성원이 트리거한 평가 → **삭제하지 않음** (다른 구성원에 대한 평가 결과이므로 보존)
- `triggered_by_user_id`의 FK는 ON DELETE CASCADE이지만, user 레코드 자체를 삭제하지 않으므로 영향 없음

### 5.6 prompt_templates — 전역 vs 프로젝트

- `project_id IS NOT NULL`인 템플릿만 삭제 (프로젝트 범위)
- `project_id IS NULL`인 전역 템플릿은 사용자 소유이므로 프로젝트 제거와 무관하게 보존

### 5.7 project_invitations (기존 기획에서 누락)

- `inviter_id = userId`인 초대 레코드도 삭제 대상
- 이미 수락된 초대는 collaborator 레코드로 전환되었으므로, 주로 만료/대기 중인 초대가 해당

### 5.8 알림

- 제거된 구성원에게 알림 발송 여부는 v1에서는 미포함 (향후 확장 가능)
