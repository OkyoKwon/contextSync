# 팀 협업 환경 개선 기획안

> 2026-03-23 · ContextSync 팀 협업 프로세스 검토 및 개선

---

## 1. 배경

ContextSync는 로컬 기반 작업 환경으로, 팀 협업 시 각 구성원이 **로컬 서버를 실행**하고 **공유 리모트 DB(Supabase)**에 연결하는 구조.
현재 팀원 온보딩과 유저 식별에 다음과 같은 문제가 있음.

---

## 2. 현재 문제점

### 2.1 유저 식별 불가 (Critical)

| 문제                               | 상세                                                                                                                                                                                      |
| ---------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Auto-login이 매번 새 유저 생성** | 앱 첫 접속 시 `POST /auth/auto`가 호출되어 `auto_<uuid>@local` 이메일로 익명 유저가 자동 생성됨. localStorage가 초기화되면(브라우저 리셋, 시크릿 모드 등) 동일인이 다른 user_id를 갖게 됨 |
| **데이터 고아화**                  | 이전 유저 ID에 연결된 세션, 프로젝트, 활동 기록에 접근 불가. 새 auto-user는 기존 프로젝트의 collaborator가 아니므로 데이터가 사실상 유실                                                  |
| **불필요한 이메일 요구**           | 로컬 환경에서 "로그인"이라는 개념 자체가 불필요. 유저 구분만 가능하면 되는데 이메일까지 요구하는 것은 과잉                                                                                |

**현재 인증 플로우:**

```
앱 로드 → localStorage에 토큰 있음?
  → Yes: 토큰으로 /auth/me 호출 → 대시보드
  → No:  /auth/auto 호출 → 익명 유저 생성 → 온보딩
```

**문제 시나리오:**

```
1. Alice가 로컬에서 앱 시작 → auto_abc@local (user_id: A) 생성
2. Alice가 프로젝트 생성, 세션 10개 축적
3. Alice가 브라우저 캐시 삭제
4. Alice가 다시 앱 시작 → auto_def@local (user_id: B) 생성
5. Alice는 이전 프로젝트/세션에 접근 불가 (user_id B는 collaborator가 아님)
```

### 2.2 온보딩 프로세스 문제

| 문제                                      | 상세                                                                                     |
| ----------------------------------------- | ---------------------------------------------------------------------------------------- |
| **유저 이름 설정이 선택적**               | auto-login 경로에서는 이름/이메일 입력 없이 바로 진입. 팀 환경에서 세션 소유자 구분 불가 |
| **팀원 합류 과정이 복잡**                 | 서버 시작 → 회원가입 → DB 설정 → Join Code 입력 등 5단계 필요                            |
| **리모트 DB 연결과 프로젝트 참여가 별개** | DB URL 입력과 Join Code 입력이 분리되어 있어 이중 작업                                   |

### 2.3 팀 가시성 부족

| 문제                      | 상세                                             |
| ------------------------- | ------------------------------------------------ |
| **세션 소유자 구분 미흡** | 세션 목록에서 누가 생성한 세션인지 명확하지 않음 |
| **동시 작업 인지 부족**   | 같은 프로젝트에서 누가 활동 중인지 알기 어려움   |

## 3. 개선안

### Phase 1: 필수 인증 강화

**목표:** 리모트 DB 환경에서 유저를 확실히 식별

#### 3.1.1 모든 환경에서 유저 이름 필수

- **DB 모드 무관** — 로컬/리모트 모두 auto-login(익명 유저 생성)을 **비활성화**
- 앱 첫 접속 시 **유저 이름만** 입력받는 간단한 화면 표시
- 이메일은 불필요 — "로그인"이 아니라 "누구인지 알려주기"만 하면 됨
- 이름 기반으로 `findOrCreateByName(db, name)` 호출하여 유저 생성/조회

**변경 플로우:**

```
앱 로드 → localStorage에 토큰 있음?
  → Yes: /auth/me → 대시보드
  → No:  /identify 페이지 → 이름 입력 → /auth/identify → 대시보드
```

**백엔드 변경:**

- 새 엔드포인트 `POST /auth/identify` — 이름만 받아서 유저 생성/조회 후 JWT 반환
- 기존 `POST /auth/login` (이메일 기반)은 유지 (하위 호환)
- 이메일은 내부적으로 `{name_slug}@local` 형태로 자동 생성 (DB unique 제약 충족)

**영향 파일:**
| 파일 | 변경 |
|------|------|
| `apps/web/src/components/auth/AppEntryRedirect.tsx` | 토큰 없으면 항상 `/identify`로 이동 (auto-login 제거) |
| `apps/api/src/modules/auth/auth.routes.ts` | `POST /auth/identify` 엔드포인트 추가 |
| `apps/api/src/modules/auth/auth.service.ts` | `findOrCreateByName()` 함수 추가 |
| `apps/web/src/pages/IdentifyPage.tsx` | 신규: 이름만 입력하는 간단한 UI |

#### 3.1.2 유저 복구 경로

- localStorage 초기화 후에도 **같은 이름을 입력**하면 기존 유저로 복구
- `findOrCreateByName`이 이름이 같으면 기존 유저 반환하도록 구현
- 추가 구현 불필요 — 인증 플로우만 변경하면 자연스럽게 해결
- 이름 충돌 시(다른 사람이 같은 이름 사용): 기존 유저 목록을 보여주고 "나입니다" 또는 "새로 만들기" 선택

#### 3.2.4 구체적 구현 계획

**Step 1: `apps/api/src/scripts/setup-team.ts` 생성**

```typescript
// 핵심 로직 pseudocode
async function main() {
  const args = parseArgs();  // node:util parseArgs
  const rl = createInterface({ input: stdin, output: stdout });

  // 1. DB URL 입력
  const dbUrl = args.dbUrl ?? await rl.question('Database URL: ');
  const ssl = args.ssl ?? true;

  // 2. 연결 테스트
  const test = await testConnection(dbUrl, ssl);
  if (!test.success) → SSL=false로 재시도 → 실패 시 에러 종료

  // 3. 마이그레이션
  await runMigrations({ connectionString: dbUrl, sslEnabled: ssl });

  // 4. .env 파일 생성 (DB 연결 후, 유저 생성 전)
  //    .env 이미 존재 시: 덮어쓸지 확인 (--force로 스킵)
  writeEnvFile({ dbUrl, ssl, jwtSecret: randomBytes(48).toString('base64') });

  // 5. DB 인스턴스 생성
  const db = createDb({ connectionString: dbUrl, ssl });
  try {
    // 6. 유저 생성
    const name = args.name ?? await rl.question('Your name: ');
    const result = await findOrCreateByName(db, name);
    const user = result.users[0]; // 신규 생성이므로 항상 1명

    // 7. Join Code (선택)
    const code = args.joinCode ?? await rl.question('Join Code (Enter to skip): ');
    if (code) {
      const project = await joinByCode(db, code.toUpperCase(), user.id);
      console.log(`Joined project: "${project.name}"`);
    }
  } finally {
    await db.destroy();
    rl.close();
  }

  // 8. 완료
  console.log('Setup complete! Run: pnpm dev');
}
```

**CLI 플래그:** `--db-url`, `--name`, `--join-code`, `--ssl` / `--no-ssl`, `--force`

**SSL 자동감지:** `ssl=true`로 시도 → 실패 시 `ssl=false`로 재시도 → 성공하면 그 값 사용

**.env 작성 필드:**

```
PORT=3001
HOST=0.0.0.0
NODE_ENV=development
DATABASE_URL={입력값}
DATABASE_SSL={true/false}
RUN_MIGRATIONS=true
JWT_SECRET={자동생성}
FRONTEND_URL=http://localhost:5173
```

**Step 3: 루트 `package.json`에 스크립트 추가**

## 5. 리스크 및 고려사항

| 항목                      | 내용                                                                                                                                                 |
| ------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- |
| **기존 auto-user 데이터** | Phase 1 적용 시, 기존 auto-user의 세션/프로젝트를 실명 유저로 마이그레이션하는 경로 필요. 기존 `POST /auth/upgrade` 엔드포인트가 이 역할을 수행 가능 |
| **로컬 전용 사용자 영향** | 로컬 DB 사용자도 최초 1회 이름 입력 필요 (이후 토큰 캐시로 재입력 불필요)                                                                            |
| **이름 충돌**             | 같은 이름의 팀원이 있을 경우 유저 선택 UI 필요. "나입니다" / "새로 만들기" 분기로 처리. 팀 규모가 작으므로 현실적으로 드문 케이스                    |
| **JWT_SECRET 공유**       | 모든 팀원이 같은 DB를 쓰더라도, 각자 다른 JWT_SECRET을 가짐. 이는 정상 — JWT는 로컬 서버에서만 검증                                                  |

## 6. 교차검수 결과

> 실제 코드 대비 기획안 검증 (2026-03-23)

| #   | 기획안 내용                    | 실제 코드 상태                                      | 대응                                                                                  |
| --- | ------------------------------ | --------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 1   | `name`으로 유저 식별           | `name`에 UNIQUE 제약 없음 (email만 UNIQUE)          | name UNIQUE 추가 시 기존 auto-user 충돌 위험 → **UNIQUE 없이 검색 후 목록 반환** 방식 |
| 2   | `/setup/status`로 DB 모드 감지 | 인증 필수 라우트 (`preHandler: [app.authenticate]`) | `/setup/status`만 **인증 없이 호출 가능하도록 분리**                                  |
| 3   | `POST /auth/identify`          | 존재하지 않음                                       | 신규 엔드포인트 구현                                                                  |
| 4   | `findOrCreateByName()`         | 존재하지 않음                                       | 신규 서비스 함수 구현                                                                 |
| 5   | IdentifyPage UI                | 존재하지 않음                                       | 신규 페이지 구현                                                                      |

### name 식별 방식 상세

DB `users.name`에는 UNIQUE 제약이 없고, 추가하면 기존 auto-user(`auto_xxx@local`)와 충돌 가능.
대안으로 이름 검색 후 결과 수에 따라 분기:

```
POST /auth/identify { name: "Alice" }
  → 0건 매치: 새 유저 생성 (이메일 자동: alice-a1b2c3d4@local) + JWT 반환
  → 1건 매치: 해당 유저로 JWT 반환 (복구)
  → 2건+ 매치: 유저 목록 반환 → 프론트에서 선택 UI
```

검색 시 `is_auto = false` 조건으로 실명 유저만 대상 (auto-user 제외).

## 7. Phase 1 구체적 구현 계획

### Step 1: `/setup/status` 인증 제거

**파일:** `apps/api/src/modules/setup/setup.routes.ts`

현재 모듈 전체에 `preHandler: [app.authenticate]` 적용됨.
`/setup/status`만 인증 없이 호출 가능하도록 라우트를 auth guard 밖으로 이동.

### Step 2: `POST /auth/identify` 백엔드

**2a. 스키마** — `apps/api/src/modules/auth/auth.schema.ts`

```typescript
export const identifySchema = z.object({
  name: z.string().min(1).max(100).trim(),
});
```

**2b. 서비스** — `apps/api/src/modules/auth/auth.service.ts`

```typescript
export async function findOrCreateByName(db: Db, name: string)
  → is_auto=false인 유저를 name으로 검색
  → 0건: 새 유저 생성 (email: {slug}-{uuid8}@local, is_auto: false)
  → 1건: 기존 유저 반환
  → 2건+: 목록 반환
  → 반환: { users: User[], created: boolean }
```

**2c. 라우트** — `apps/api/src/modules/auth/auth.routes.ts`

```
POST /auth/identify    → name 받아서 findOrCreateByName 호출 → JWT 반환 또는 목록 반환
POST /auth/identify/select → userId 받아서 JWT 발급 (동명이인 선택용)
```

두 엔드포인트 모두 인증 불필요 (로그인 전 호출).

### Step 3: 프론트엔드 API

**파일:** `apps/web/src/api/auth.api.ts`

```
identify(name: string) → POST /auth/identify
identifySelect(userId: string) → POST /auth/identify/select
```


  - 유저 2명+: 목록 표시 → 선택 또는 "새로 만들기"
