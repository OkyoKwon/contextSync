# Landing Page / App 분리 계획

## 배경

현재 `/` 경로가 마케팅용 랜딩 페이지를 보여준다. 로컬에서 개인 용도로 사용하는 유저에게는 불필요한 마찰이다.
랜딩 페이지는 외부 공개용(GitHub Pages, Vercel 등)으로 분리하고, 로컬 앱은 `/`에서 바로 대시보드로 진입하도록 변경한다.

**현재 흐름:**

```
로컬 유저:  / (랜딩) → 시작하기 클릭 → /onboarding → /dashboard
외부 방문:  / (랜딩) → 시작하기 클릭 → /onboarding → /dashboard
```

**목표 흐름:**

```
로컬 유저:  / → 자동 로그인 → /dashboard (랜딩 없음)
외부 방문:  / (랜딩) → 시작하기 클릭 → /onboarding → /dashboard
```

---

## 핵심 결정: 어떻게 분리할 것인가

### 방안 비교

| 방안              | 설명                                  | 장점                            | 단점                                     |
| ----------------- | ------------------------------------- | ------------------------------- | ---------------------------------------- |
| A. 환경변수 분기  | `VITE_LANDING=true` 일 때만 랜딩 표시 | 단일 코드베이스, 간단           | 빌드에 랜딩 코드 포함                    |
| B. 별도 앱 분리   | `apps/landing/` 새 Vite 앱            | 완전 분리, 독립 배포            | 공유 컴포넌트 관리, 모노레포 복잡도 증가 |
| C. 빌드 타임 분리 | 두 개의 Vite 엔트리포인트             | 같은 디렉토리, 별도 빌드 산출물 | 빌드 설정 복잡                           |

### 선택: 방안 A (환경변수 분기)

이유:

- 최소 변경으로 구현 가능
- 기존 `DEPLOYMENT_MODE` 환경변수(`personal` | `team-host` | `team-member`)와 자연스럽게 연동
- 랜딩 페이지 컴포넌트들이 lazy-loaded이므로 로컬 앱 번들에 영향 없음
- 추후 방안 B로 점진적 마이그레이션 가능

## 구현 계획

### Phase 1: 프론트엔드 — 라우팅 분기

#### 1-1. 환경변수 추가

**수정**: `apps/web/.env.example`, `apps/web/.env` (해당 시)

```
# true면 / 경로에서 랜딩 페이지 표시 (외부 공개용)
# false면 / 경로에서 바로 앱 진입 (로컬 개인용)
VITE_SHOW_LANDING=false
```

#### 1-2. 라우팅 분기

**수정**: `apps/web/src/routes.tsx`

`/` 경로의 동작을 환경변수로 분기:

```typescript
const SHOW_LANDING = import.meta.env.VITE_SHOW_LANDING === 'true';
```

- `SHOW_LANDING=true`: 기존처럼 `<LandingPage />` 렌더링
- `SHOW_LANDING=false`: `<AppEntryRedirect />` 렌더링
  - 토큰 있으면 → `/dashboard`로 리다이렉트
  - 토큰 없으면 → auto-login 실행 후 → `/onboarding` 또는 `/dashboard`

#### 1-3. AppEntryRedirect 컴포넌트

**새 파일**: `apps/web/src/components/auth/AppEntryRedirect.tsx`

```
1. useAuthStore에서 token 확인
2. token 있음 → <Navigate to="/dashboard" />
3. token 없음 → authApi.autoLogin() 호출
   - 성공 → setAuth → onboarding 상태 확인 → /dashboard 또는 /onboarding
   - 실패 → /login으로 fallback
4. 로딩 중 → 스피너 표시
```

이 컴포넌트는 방금 구현한 auto-user 기능을 재사용한다.

#### 1-4. LoginPage 조건부 "홈으로" 링크

**수정**: `apps/web/src/pages/LoginPage.tsx`

- `SHOW_LANDING=true`: "← 홈으로" 링크 표시 (현재)
- `SHOW_LANDING=false`: "← 홈으로" 링크 숨김 (대시보드가 홈이므로 의미 없음)

---

### Phase 2: 랜딩 전용 배포 설정

#### 2-1. 랜딩용 환경 파일

**새 파일**: `apps/web/.env.landing`

```
VITE_SHOW_LANDING=true
VITE_API_TARGET=https://api.contextsync.dev  # 또는 공개 API 주소
```

#### 2-2. 빌드 스크립트

**수정**: `apps/web/package.json`

```json
{
  "scripts": {
    "build": "tsc && vite build",
    "build:landing": "tsc && vite build --mode landing"
  }
}
```

`--mode landing`을 사용하면 Vite가 `.env.landing` 파일을 자동 로드한다.

#### 2-3. 로컬 기본값

`apps/web/.env`에서 `VITE_SHOW_LANDING=false`를 기본값으로 설정.
로컬 `pnpm dev` 실행 시 랜딩 없이 바로 앱 진입.

---

### Phase 3: /docs, /login 경로 처리

랜딩 페이지가 숨겨져도 `/docs`와 `/login`은 독립적으로 접근 가능해야 한다.

#### 3-1. /docs 유지

변경 없음. 이미 독립 라우트로 존재.

#### 3-2. /login 유지

변경 없음. 팀 합류, 기존 계정 로그인 시 사용.
`SHOW_LANDING=false`일 때는 직접 URL 입력으로만 접근 (네비게이션에서 노출하지 않음).

#### 3-3. LandingNav / LandingFooter 스코프

`SHOW_LANDING=false`일 때 이 컴포넌트들은 `/` 경로에서 사용되지 않으므로 로드되지 않음.
`/docs` 페이지가 자체 네비게이션을 사용하므로 영향 없음.

---

### Phase 4: i18n

#### 4-1. 번역 키 추가

**수정 파일**: `types.ts`, `ko.ts`, `en.ts`, `ja.ts`

| 키            | ko         | en         | ja            |
| ------------- | ---------- | ---------- | ------------- |
| `app.loading` | 로딩 중... | Loading... | 読み込み中... |

기존 랜딩 관련 키는 유지 (랜딩 모드에서 여전히 사용).

## 검증 방법

1. **로컬 기본 모드** (`VITE_SHOW_LANDING` 미설정 또는 `false`):
   - `pnpm dev` → 브라우저에서 `/` 접속
   - 랜딩 페이지 없이 자동 로그인 → 대시보드 진입 확인
   - 이미 로그인된 상태면 바로 `/dashboard` 리다이렉트

2. **랜딩 모드** (`VITE_SHOW_LANDING=true`):
   - `.env`에 `VITE_SHOW_LANDING=true` 설정 후 `pnpm dev`
   - `/` 접속 시 기존 랜딩 페이지 정상 표시
   - "시작하기" 클릭 → auto-login → onboarding 흐름 동작

3. **빌드**:
   - `pnpm build` → 로컬 앱 빌드 (랜딩 없음)
   - `pnpm --filter @context-sync/web build:landing` → 랜딩 포함 빌드

4. **기존 테스트 호환**:
   - E2E 테스트는 `VITE_SHOW_LANDING=true` 환경에서 실행 (기존 랜딩 테스트 유지)
   - `pnpm test && pnpm typecheck` 통과

## 향후 확장

- **방안 B 마이그레이션**: 랜딩 페이지를 `apps/landing/`으로 완전 분리하여 GitHub Pages나 Vercel에 독립 배포
- **커스텀 도메인**: 앱은 `app.contextsync.dev`, 랜딩은 `contextsync.dev`로 분리
- **DEPLOYMENT_MODE 연동**: 백엔드 `DEPLOYMENT_MODE=personal`일 때 프론트엔드에 힌트를 내려줘서 환경변수 없이도 자동 판단
